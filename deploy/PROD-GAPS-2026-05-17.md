## Remaining hard fails

### 1. Storage — `NoCredentialsError: Unable to locate credentials`

**Diagnosis**: env var `S3_ACCESS_KEY_ID` and/or `S3_SECRET_ACCESS_KEY`
are NOT set on the api Railway service. The boto3 default provider
chain (IAM role, ~/.aws/credentials, env vars) has nothing to grab.

**Operator action**:

```
Railway dashboard → aec-platform-api → Variables → Add:
  S3_ACCESS_KEY_ID       = <access key>
  S3_SECRET_ACCESS_KEY   = <secret key>
  S3_BUCKET              = aec-platform-files       # already set
  AWS_REGION             = ap-southeast-1           # default
  S3_ENDPOINT_URL        = (leave EMPTY for AWS S3)
                        = http://<minio-host>:9000  # for MinIO
```

Same env vars must also be set on `aec-platform-worker` (the worker
writes drawing chunks + photo thumbs + report PDFs).

**Verify**: re-run `make verify-deploy AEC_BASE_URL=…` → check #4
should flip to `✓ Storage bucket reachable (aec-platform-files)`.

### 2. CodeGuard regulations table still empty

**Possible causes** (in priority order):

1. **Worker not redeployed yet**. The L8 changes (start cmd alembic
   run) shipped — but the worker has its own Railway service +
   railway.worker.json config. Check Railway dashboard for the worker:
   does it have the recent commit deployed?

2. **Worker missing `GOOGLE_API_KEY`**. The bootstrap function:
   ```python
   # workers/codeguard_bootstrap.py
   if not os.getenv("GOOGLE_API_KEY"):
       logger.warning("codeguard_bootstrap: ... GOOGLE_API_KEY not set — skipping")
       return
   ```

   Silent skip if the key isn't on the worker. Operator must set it.

3. **`CODEGUARD_BOOTSTRAP_DISABLED=1` set** by accident.

**Operator action**:

```

  GOOGLE_API_KEY                = AIza...

```

After confirming, **restart the worker service** (or wait for next
deploy). On next worker boot, look in logs for:

```
…
```

If you see `disabled via env var, skipping` → check #3.
