## Hard fails (fix before announcing launch)

### 1. Migration drift — alembic_version stuck at `c16cdbca5b41`

**Root cause**: Railway api service startCommand was `uvicorn …` only.
Migrations never auto-ran on deploy. Migrations 0050 (billing), 0051
(llm_spend), 0052 (cashflow), 0053 (safety_toolbox), 0054
(subcontractor_portal) are NOT applied in prod.

**Fix shipped** (commit pending): `railway.json` startCommand now runs
`alembic upgrade head` before `uvicorn`. Pipes `set -e` so failure
fails the deploy loud (Railway marks as failed instead of running
old code silently).

**Manual one-time**: after this commit redeploys, the start command
will run all pending migrations. ~30 seconds added to cold start
of first deploy; subsequent deploys are no-op when head matches.

**Verify after deploy**: `/health/ready.checks.migration.head` should
equal `0054_subcontractor_portal`.

### 2. Storage (S3/MinIO) timeout

**Symptom**: `checks.storage.error = "timeout (>3s)"` (after my
bump from 2s). `configured: true` means `S3_BUCKET` env IS set on
Railway.

**Likely causes** (in priority order):
1. `S3_ENDPOINT_URL` points at MinIO that's not actually deployed
   (the repo has docker-compose for MinIO; prod might have S3_BUCKET
   set but no actual storage backend).
2. `AWS_REGION` mismatch with where the bucket lives.
3. `S3_ACCESS_KEY_ID` / `S3_SECRET_ACCESS_KEY` wrong.
4. Bucket name typo (the S3_BUCKET value doesn't match any bucket
   the credentials can see).

**Manual action**:
* Railway dashboard → aec-platform-api → Variables → audit:
  - `S3_BUCKET`
  - `S3_ENDPOINT_URL` (should be empty for AWS S3, OR
    `https://your-minio.host` for MinIO)
  - `AWS_REGION` (default `ap-southeast-1`)
  - `S3_ACCESS_KEY_ID` + `S3_SECRET_ACCESS_KEY`
* Quick smoke: SSH into Railway service + `aws s3 ls s3://$S3_BUCKET --region $AWS_REGION` should return the bucket without error.

### 3. Redis intermittent timeout

**Symptom**: `/health/ready.checks.redis` flapping between ok + timeout
on consecutive probes. Same readiness probe call's check #3
(via `/metrics`) shows `arq_queue_depth = 0.0` (Redis reachable),
but the readiness sub-probe times out.

**Likely cause**: cold connection. Each readiness call instantiates
a fresh arq pool via `create_pool` → TLS handshake to Upstash. First
call after idle is slow; subsequent calls within the connection
TTL are fast.

**Fix shipped**: bumped readiness Redis timeout 1s → 2s.

**Longer-term fix** (deferred): readiness probe should reuse a
process-global Redis pool instead of creating a fresh one on each
probe. Adds ~100 LOC; out of scope for this iteration.

### 4. CodeGuard regulations table empty

**Symptom**: `count = 0` in `/health/ready.checks.codeguard_regulations`.

**Root cause**: chained from #1. The bootstrap function
(`apps/api/workers/codeguard_bootstrap.py`) runs at worker startup,
checks `SELECT COUNT(*) FROM regulations` — but the regulations
table doesn't exist yet because migration 0005 (codeguard) hasn't
been run.

**Fix**: After #1 is fixed and migrations run, the next worker
restart will trigger the bootstrap. Bootstrap also needs
`GOOGLE_API_KEY` set on the worker service (see warnings below).

**Manual verify**: after migrations + worker restart, the bootstrap
log line `codeguard_bootstrap: ingested QCVN 06:2022/BXD — 13 sections,
13 chunks` should appear in worker logs.

## Warnings — operator action required (no code fix)

### W1. SSO providers not toggled

Railway env vars don't show `SUPABASE_URL` from this verify shell
(local limitation), but you need to verify these on Supabase
dashboard:

1. Supabase project dashboard → **Authentication → Providers**
2. Toggle **Google** ON — paste OAuth Client ID + Secret from
   Google Cloud Console.
3. Toggle **Azure (Microsoft)** ON — paste Application ID + Secret
   from Entra portal.
4. Supabase → **Authentication → URL Configuration → Redirect URLs**:
   Add `https://aec-platform-web.vercel.app/auth/callback`
   + `https://aec-platform-web.vercel.app/auth/callback?next=/**`

Full runbook: `deploy/SSO-SETUP.md`

### W2. Email backend (Resend) not configured

Worker can't send Pulse weekly reports, RFQ supplier dispatch,
codeguard quota threshold alerts, or invitation emails.

1. Sign up at https://resend.com.
2. Verify domain `aec-platform.vn` (3 CNAME records they generate).
3. Generate API key.
4. Railway → aec-platform-worker → Variables → add `RESEND_API_KEY`
   + `RESEND_FROM=AEC Platform <no-reply@aec-platform.vn>`.

### W3. Stripe not configured

Only VietQR billing works. International card customers (foreign-
owned EPCs, Stripe demos) cannot pay via card.

1. Create Stripe products: "Chuyên nghiệp Monthly USD" ($199).
2. Set webhook endpoint:
   `https://aec-platform-production.up.railway.app/api/v1/billing/webhooks/stripe`
   with events `checkout.session.completed`, `customer.subscription.deleted`,
   `invoice.payment_failed`.
3. Railway env: `STRIPE_SECRET_KEY` + `STRIPE_WEBHOOK_SECRET`.

### W4. Sentry not configured

Errors silently drop into Railway logs. No alerts, no issue grouping,
no release tracking.

1. Sign up at https://sentry.io. Create project `aec-platform`.
2. Railway api + worker env: `SENTRY_DSN=https://...@.../...`
3. Optional: `SENTRY_TRACES_SAMPLE_RATE=0.1`, `SENTRY_PROFILES_SAMPLE_RATE=0`.

Full runbook: `deploy/OBSERVABILITY.md`

### W5. GOOGLE_API_KEY not visible from outside

Can't verify from a public probe. Critical for:
* CodeGuard bootstrap (needs Gemini embedding to ingest QCVN
  excerpts)
* Drawbridge ingest (needs Gemini embedding for chunks)
* Pulse weekly report (needs Gemini for narrative summary)

**Manual verify**: Railway → aec-platform-worker → Variables → confirm
`GOOGLE_API_KEY` starts with `AIza...`.
